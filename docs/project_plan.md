# Project Plan — Retinal Foundation Model Pipeline

## Purpose of this document

This is the operational plan for executing the project. It sits alongside the four primary documents (specification, implementation reference, architecture guardrails, issues-and-solutions log) and tells the reader and any AI tools how the project is being built, in what sequence, with what tools, and under what discipline. The four primary documents define what is being built and what the rules are. This document defines how the building happens.

## Source-of-truth precedence

When documents conflict, later documents override earlier ones. The order is:

1. Project specification — the what and why
2. Implementation reference — concrete numerical defaults and gotchas
3. Architecture guardrails — 76 hard rules
4. Issues-and-solutions log — resolved edge cases
5. Decisions log — running record of choices made during implementation

This precedence is not optional. Any tool reading the project files must apply it. The decisions log is the most current source of truth for any item it covers.

## Workflow: hybrid generation

The codebase is produced through a three-phase workflow.

**Phase 1: Architecture and load-bearing files generated in conversation with Claude (Opus).** This phase produces the directory structure, the AI context pack, the canonical schema, the enums module, the task registry, the feature policy, the DatasetAdapter base class, the DummyAdapter, and the patient-level split utility with its audit test. These are the contracts that every subsequent file must respect. They are generated one file at a time, with explicit review after each, to ensure architectural consistency.

**Phase 2: MVP fill-in via Claude Code (Opus for architectural stages, Sonnet for fill-in).** Claude Code takes the Phase 1 skeleton plus the four primary documents plus the AI context pack and fills in the remaining MVP files: ODIRAdapter, preprocessing pipeline, embedding extraction, multi-task head, training loop, evaluation harness. Claude Code runs the dummy-pipeline end-to-end test and the ODIR pipeline end-to-end test before the phase is considered complete.

**Phase 3: Refinement and feature additions via Claude Code.** All post-MVP work happens here: external dataset adapters, fairness mitigation, LoRA with replay buffer and Mahalanobis OOD detection (PCA-64 variant per issues log), continual-learning simulation, explainability stack, deployment dashboard, multi-seed runs, and the BRSET adapter when access is granted. Each feature is a constrained addition to a working system.

## Tool division of labour

**Conversation with Claude (Opus, web UI):**

- Final architecture design and architectural amendments
- File-by-file specifications
- High-risk implementation files (schema, task registry, feature policy, adapter base class, split utility, multi-task head)
- Reviews of generated code for guardrail compliance
- Debugging architectural errors
- Writing exact prompts for Claude Code and Codex
- Resolving design questions not pinned down in the four documents

**Claude Code (Opus for architectural stages, Sonnet for fill-in):**

- Applying patches across the repo
- Creating files from a given file specification
- Running tests and verification scripts
- Fixing import errors and routine bugs
- Refactoring after the target is defined in conversation
- Generating repetitive boilerplate (external dataset adapters once the pattern is set)
- Default to Opus when in doubt; the cost difference is small compared to the cost of a subtle architectural bug

**Codex / Gemini Antigravity / IDE agents:**

- Cross-tool audit (passing files generated by one tool to another with prompts like "audit for dataset coupling violations")
- Repo navigation and local autocomplete
- Minor edits and environment commands
- Inspecting file relationships

**What no tool may do without explicit review:**

- Modify the canonical schema, task registry, feature policy, adapter base class, or split utility
- Modify the evaluation protocol or fairness harness once frozen
- Modify the continual-learning protocol rules
- Add dataset-specific code outside an adapter
- Change preprocessing for an existing backbone
- Change the four-way split ratios or the patient-grouping discipline

## AI context pack

A directory at `docs/ai_context/` contains compressed, machine-readable documents that any AI tool reads before touching the repo. These are written by the architecture controller (Claude Opus, in conversation) once the blueprint is finalised. They are intentionally redundant with the four primary documents — the redundancy is the point, because each tool gets a short, precedence-clear summary instead of having to re-derive conventions from full PDFs.

The pack contains:

- `00_source_of_truth_order.md` — the precedence rule stated explicitly
- `01_architecture_contract.md` — ten-layer architecture, dataset-coupling status per layer, the BRSET-swap promise condensed
- `02_guardrails_compressed.md` — the 76 guardrails distilled to the ~20 hard rules with the hard/soft/aspirational categorisation
- `03_file_generation_order.md` — the build order with verification gates after each stage
- `04_forbidden_patterns.md` — explicit anti-patterns (no `if dataset == "odir"` outside adapters, no image-level splits, no augmentation in val/test, no using age as input when retinal age is target, no silent NaN reporting on sparse subgroups)
- `05_dataset_adapter_contract.md` — the exact interface every adapter must implement, with the canonical schema and per-task supported-list specification
- `06_testing_protocol.md` — when the dummy pipeline is run, what tests must pass before each stage moves forward

A `PROTECTED_FILES.md` at the repo root lists files that any tool must flag before modifying. The list includes the canonical schema, task registry, feature policy, adapter base class, split utility, evaluation harness contracts, and continual-learning protocol rules.

## Decisions log

A `docs/decisions.md` file is appended whenever a design question is resolved during implementation. Each entry records date, decision, rationale, and the file or files affected. This protects against context drift across conversations and across tools — when a new conversation starts, the log shows what is already settled.
